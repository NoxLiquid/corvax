## Overview

Вирусология — полноценный субдепартамент медицинского блока, добавляющий систему инфекционных заболеваний с возможностью их исследования, эволюции и лечения. Система включает роль вирусолога, набор лабораторного оборудования, линейку противовирусных препаратов, механику разумного заболевания как антагониста и события эпидемий разной сложности.

---

## Background

В базовом SS14 (и Corvax) отсутствует полноценная система заболеваний. Медицинский блок работает исключительно с травмами (лечением урона), но не с инфекциями. Т.е. у врачей нет долгосрочных задач, связанных с исследованием и профилактикой.

Вирусология решает эти проблемы: даёт врачам новую роль, а науке интеграцию с вирусологической лабораторией через научные очки, а раунду больше уникальности.

## Features to be added

### 1. Данные заболевания (DiseaseData)

Каждое заболевание представлено структурой `DiseaseData`, наследующей `ReagentData`. Это позволяет хранить вирус непосредственно в растворах (как реагент) и использовать стандартную инфраструктуру химии SS14.

Ключевые параметры штамма:

- **StrainId** — уникальный 6-символьный буквенно-цифровой идентификатор штамма, генерируемый автоматически.
- **Threshold / MaxThreshold** — текущая и максимальная живучесть вируса (по умолчанию 100). Когда Threshold падает до 0, заболевание считается вылеченным.
- **RegenThreshold** — скорость регенерации живучести за тик (по умолчанию 1).
- **Infectivity** — базовая заразность (от 0 до 1). Увеличивается симптомами.
- **MutationPoints / RegenMutationPoints** — очки мутации и скорость их регенерации. Тратятся на покупку симптомов и расширение "белого списка" организмов (сущностей, которых может заразить рассматриваемое заболевание).
- **MedicineResistance** — словарь устойчивости к конкретным препаратам. Растёт при каждом применении лекарства (антибиотика).
- **DefaultMedicineResistance** — базовый уровень устойчивости ко всем препаратам.
- **ActiveSymptom** — список активных симптомов.
- **SpeciesWhitelist** — список рас, которые может заражать вирус.
- **EntityWhitelist** — стандартный белый список компонентов (MobState + Bloodstream).
- **Color** — цвет штамма (случайный при генерации), используется для визуализации облаков и загрязнения.
- **SpeedModifier** — модификатор скорости вируса.
- **MultiPriceDeleteSymptom** — множитель стоимости удаления симптома
- **DamageWhenDead** — урон получаемый вирусом с течением времени, если его носитель мёртв.

### 2. Компонент заболевания (DiseaseComponent)

Навешивается на заражённую сущность. Содержит:

- Ссылку на `DiseaseData` штамма.
- Список активных экземпляров симптомов (`ActiveSymptomInstances`).
- Состояние носителя (`PatientState`).
- Тип кровати, к которой привязан пациент (`RegenerationType`).
- Таймер обновления (1 секунда).

### 3. Механика заражения

Заражение происходит тремя путями:

- **Воздушно-капельный** — облака инфекции (DiseaseInfectionCloud) распространяются в пространстве через систему EdgeSpreader. При коллизии с мобом происходит попытка заражения.
- **Контактный** — контаминация (DiseaseContaminationComponent) передаётся при столкновении с облаком и затем распространяет инфекцию вокруг заражённого объекта.
- **Через раствор** — реагент `DiseaseSolution` при метаболизме в крови или в чаше Петри вызывает заражение.

**Формула шанса заражения:**
```
Шанс = Заразность − Суммарный резист одежды
Результат ∈ [0%, 100%]
```

Резист складывается из модификаторов экипировки (слоты: ноги, голова, глаза, перчатки, маска, шея, нижняя и верхняя одежда).

**Конкуренция штаммов:** заболевание с бо́льшим количеством симптомов вытесняет более слабое. Это позволяет использовать безобидные вирусы с большим числом симптомов как "вакцину".

**Иммунитет:** при излечении у сущности появляется `DiseaseImmunComponent` со списком штаммов, к которым выработан иммунитет.

**Ограничения заражения:**

Константные правила модуля (чёрный список рас, белый список компонентов, стоимости операций, модификаторы кроватей и др.) определяются в `Content.Shared/_Modifications/Disease/SharedDiseaseSystem.cs` (структура `BaseDiseaseSettings`). При интеграции в сборку этот файл необходимо изучить и адаптировать под свой контент — в частности, скорректировать `BodyBlackList` (расы, невосприимчивые к заражению), `DefaultWhitelistComponents` (требуемые компоненты у цели) и ценовые константы (`StaticBodyPrice`, `StaticPriceDeleteSymptom` и т.д.). Аналогично, логика проверок в `DiseaseSystem.CanInfect` определяет, какие состояния сущности блокируют заражение — при добавлении новых механик (аналогов зомби, новых состояний мобов) эти проверки потребуют расширения.

### 4. Облака инфекции

Сущность `DiseaseInfectionCloud`, видимая через специальные очки вирусолога (через кастомный visibility layer). Несёт клон данных штамма. Распространяется через систему `EdgeSpreader` на соседние тайлы (настраиваемое количество spread-шагов). При коллизии с мобом пытается заразить и оставляет контаминацию.

### 5. Система симптомов

Каждый симптом реализует интерфейс `IDiseaseSymptom` и наследует `DiseaseSymptomBase`. Регистрация симптомов — автоматическая: при инициализации `DiseaseSystem` сканирует все сборки на наличие классов с атрибутом `[DiseaseSymptom("ProtoId")]` и создаёт реестр фабрик.

Жизненный цикл симптома:
- `OnAdded` — при добавлении (запускает `ApplyDataEffect(true)` для модификации данных штамма).
- `OnUpdate` — периодически (по таймеру, с учётом модификатора кровати). Вызывает `DoEffect`.
- `OnRemoved` — при удалении (откатывает `ApplyDataEffect(false)`).
- `ApplyDataEffect` — изменяет параметры DiseaseData (заразность, регенерацию, резистентность и т.д.).

Если симптомы не могут проявиться (чаша Петри, первичный пациент, зомби, мертвец), вызывается только `ApplyDataEffect` без `DoEffect`.

**Категории симптомов по уровню опасности (`DangerIndicatorSymptom`):**

Каждый симптом обязательно помечается одной из четырёх категорий опасности: `Low`, `Medium`, `High`, `Cataclysm`. Категория — это не просто метка для документации, она используется в двух ключевых местах:

1. **Генерация случайных эпидемий** (`DiseaseSystem.GenerateDiseaseData`) — события эпидемий задают конфигурацию вируса в формате "сколько симптомов какой категории взять" (например, `Low: 4, Medium: 3, High: 1`). Генератор случайным образом выбирает из пула симптомов соответствующей категории (с учётом флага `CanGenerateRandomly`). Это позволяет через правила событий (`DiseaseOutbreakRuleComponent`) точно контролировать сложность эпидемии, не перечисляя конкретные симптомы.

2. **Фильтрация в UI консоли эволюции** — симптомы категории `Cataclysm` скрыты из интерфейса покупки в консоли эволюции вирусолога. Они не доступны для ручной модификации через лабораторное оборудование и предназначены исключительно для разумного вируса-антагониста или прямого назначения через код/прототипы.

Таким образом, категория определяет: в каких событиях может появиться симптом и доступен ли он вирусологу для модификации. Конкретный набор симптомов в каждой категории зависит от сборки — определяется прототипами в `Resources/Prototypes/_Modifications/Disease/symptoms.yml`.

Каждый симптом имеет цену покупки. Фактическая цена = `max(1, текущее кол-во симптомов) × базовая цена`, что делает наращивание каждого следующего симптома всё дороже.

### 6. Лечение

Каждое заболевание имеет живучесть (Threshold). Антибиотики наносят урон вирусу, снижая его. При Threshold ≤ 0 — излечение.

**Устойчивость к лекарствам:** с каждым применением конкретного антибиотика увеличивается сопротивление к нему. Формула финального урона:
```
Урон = baseDamage × max(0, 1 − resistance)
```
Это вынуждает чередовать препараты.

**Как устроен антибиотик:**

Рагент с эффектом `DamageDiseaseEffect`. Этот эффект определяется в `Content.Shared/_Modifications/Disease/Effects/DamageDiseaseEffect.cs` и имеет три атрибута:

| Атрибут | Тип | По умолч. | Назначение |
|---------|-----|-----------|------------|
| `baseDamage` | float | 1 | Базовый урон, наносимый вирусу за тик метаболизма. Масштабируется через `args.Scale`. |
| `resistanceIncrease` | float | 0.05 | На сколько увеличивается устойчивость вируса к этому ключу при каждом применении. Чем выше значение — тем быстрее препарат теряет эффективность. |
| `key` | string | `"базовый"` | Ключ резистентности. Вирус запоминает сопротивление отдельно для каждого ключа. Если у двух реагентов одинаковый `key`, устойчивость к ним общая. Обычно совпадает с ID реагента, чтобы резист рос только от конкретного препарата. |

При срабатывании эффекта (`DamageDiseaseEntityEffectsSystem`) вызывается `DiseaseSystem.ApplyMedicineDamage`, который наносит урон вирусу с учётом текущей резистентности и увеличивает её.

**Пример прототипа антибиотика:**
```yaml
- type: reagent
  id: Infectizine
  metabolisms:
    Bloodstream:
      effects:
      - !type:DamageDiseaseEffect
        baseDamage: 5
        resistanceIncrease: 0.01
        key: Infectizine
```

Конкретный набор антибиотиков, их рецепты и побочные эффекты (токсичность при передозировке) может зависеть от сборки — см. `Resources/Prototypes/_Modifications/Disease/Medicine/reagents.yml` и `reaction.yml`.

**Антисептик** — реагент контактного действия (`ReactiveGroup: Antiseptic`). При внешнем контакте (Touch) уничтожает заражение на поверхности сущности. При попадании внутрь наносит химический ожог.

**Кровати:** лежание на обычной кровати снижает регенерацию вируса вдвое (множитель 0.5), на стазис-кровати — в 10 раз (множитель 0.1). Одновременно замедляется проявление симптомов.

### 7. Оборудование вирусолога

**Портативное:**
- **Анализатор заболеваний** (HandheldDiseaseAnalyzer) — портативный прибор на батарейке, показывает окно с данными о штамме.
- **Очки обнаружения заболеваний** (DiseaseDetectorEyesGlasses) — позволяют видеть облака инфекции и визуально определять заражённых.
- **Коллектор ДНК-материала** (DiseaseDataCollector) или же стерильная палочка — одноразовый сборщик биологических данных у заражённого. Вставляется в диагност.

**Стационарное (связывается сигнальными портами через мультитул):**
- **Консоль диагноста** — центральный пульт. Функции: сканирование заболевания, печать отчёта, генерация раствора, запуск анализа раствора, удаление штамма с сервера. Связывается с диагностом (≤4 тайла), сервером данных (≤50 тайлов), анализатором веществ (≤4 тайла).
- **Диагност заболеваний** — машина с двумя слотами: ДНК-коллектор и колба. Сканирует образец и генерирует раствор с заболеванием.
- **Сервер данных** — хранилище штаммов. Приносит пассивный доход научных очков (за каждый симптом ×2 очка + за каждое тело ×1 очко). Хранит очки мутации для консоли эволюции.
- **Анализатор веществ** (DiseaseSolutionAnalyzer) — извлекает данные о заболевании из раствора в колбе.
- **Консоль эволюции** — позволяет модифицировать заболевания: покупать/удалять симптомы и типы организмов. Расходует очки мутации с сервера данных. Связывается с сервером данных и анализатором веществ.

**Чаша Петри** — лабораторный сосуд (10 единиц объёма) для культивирования заболеваний вне живого организма. Симптомы не проявляются внутри чаши, но вирус эволюционирует (очки мутации тратятся). Чаша из стекла — при разрушении распространяет заражение.

**Всё оборудование (кроме портативного) исследуется через технологию "Вирусология".**

### 8. Роль вирусолога

Новая должность в медицинском отделе. Доступы: Medical, Virology, Maintenance. Имеет собственный КПК и ID-карту.

### 9. Разумное заболевание (антагонист)

Особый антагонист — бестелесный призрак-вирус (сущность `SentientDisease`), управляемый игроком через ghost-роль. Базовые характеристики:
- Высокая скорость перемещения (walk: 20, sprint: 10).
- Проходит через стены (Incorporeal).
- Не может говорить.
- Начальные очки мутации: 3000.
- Видит через стены (drawFov: false, pvs: 1.5).

**Механики разумного вируса:**
- **Выбор первичного пациента** (EntityTargetAction) — заражает цель, превращая её в первичного пациента. Максимум 3 первичных пациента. Первичный пациент автоматически распространяет инфекцию вокруг себя каждые 1-5 секунд в радиусе 2 тайлов.
- **Телепортация к первичному пациенту** — циклический выбор между заражёнными для быстрого перемещения.
- **Магазин мутаций** — встроенный интерфейс эволюции (тот же DiseaseEvolutionConsole, но интегрированный в сущность). Покупка/удаление симптомов и организмов.
- **Компенсация** — если первичный пациент уходит в криосон, разумный вирус получает 5000 очков мутации обратно, но теряет пациента.
- Очки мутации генерируются по тику (базово 1/тик, увеличивается симптомом "эволюционный всплеск").

Пресет раунда с разумным вирусом (`SentientDiseaseRule`) существует, но закомментирован. Разумный вирус может спавниться как событие (SentientDiseaseOutbreakRule: вес 3, минимум 35 игроков, начало с 45 минуты).

### 10. События эпидемий

Три уровня случайных эпидемий как станционных событий:

| Уровень | Симптомы | Тела | Вес | Старт | Рекуренция | Мин. игроков |
|---------|----------|------|-----|-------|------------|--------------|
| Low | 2 Low + 1 Med | 4 | 10 | 15 мин | 10 мин | 10 |
| Medium | 4 Low + 3 Med | 7 | 8 | 30 мин | 15 мин | 10 |
| Hard | 4 Low + 3 Med + 1 High | 9 | 6 | 60 мин | 20 мин | 10 |

При активации события случайно генерируется вирус с указанным набором симптомов и заражается 1 случайный игрок (первичный пациент).

## Game Design Rationale

### Осмысленные выборы

**Для вирусолога:**
- Какой антибиотик применить? Слабые менее токсичны, но медленнее; сильные эффективнее, но могут отравить пациента. Чередование обязательно из-за резистентности.
- Исследовать вирус (пассивный доход науки) или немедленно уничтожить?
- Создать "вакцинный" вирус с безобидными симптомами, чтобы защитить экипаж через конкуренцию штаммов?

**Для разумного вируса:**
- Тратить очки на дешёвые Low-симптомы для быстрого распространения (кашель = облака) или копить на High/Cataclysm для летальности?
- Когда расширять белый список организмов (сущностей, которые подвержены заражению)? Каждое новое тело стоит дороже, но увеличивает пул потенциальных носителей.
- Как распределять первичных пациентов?

### Риск vs. награда

- Некровир (урон 25) — самый эффективный антибиотик, но токсичен при 4u и наносит клеточный урон. Использование — последнее средство.
- Зомбификация (3000 очков) — катастрофическое оружие, но стоит больше, чем начальный запас разумного вируса. Нужна значительная эволюция прежде чем её купить.
- Чаша Петри позволяет безопасно исследовать вирус, но при разрушении (случайном или диверсионном) выпускает его наружу.

### Кооперация

- Вирусолог зависит от химиков (рецепты антибиотиков), науки (исследование технологии Вирусологии), и безопасности (карантин).
- Эпидемия — общестанционная проблема: экипаж должен использовать защитную одежду, избегать скоплений, помогать медикам с доставкой пациентов.

### Соответствие Core Design

- **"The game should be fun for everyone"** — разумный вирус создаёт долгоиграющую угрозу, а не мгновенный вайп. Экипаж может сопротивляться, медики получают содержательную работу.
- **"Roles should be meaningful"** — вирусолог — не просто "другой доктор", а специалист со своим оборудованием, задачами и навыками.
- **"Antags create stories, not just chaos"** — разумный вирус выстраивает нарратив: от невинного кашля до катастрофы, с видимой эскалацией.

## Roundflow & Player interaction

### Стадии эпидемии в раунде

1. **Инкубация (5-30 мин):** событие эпидемии заражает 1 случайного игрока. Симптомы начинают проявляться (кашель, сыпь). Облака начинают распространяться.
2. **Обнаружение:** экипаж замечает кашель/рвоту. Вирусолог диагностирует пациента анализатором, собирает ДНК-коллектором образец, загружает в диагност. Данные сохраняются на сервер.
3. **Исследование и лечение:** вирусолог подбирает антибиотик, укладывает тяжёлых на стазис-кровать. Химик синтезирует нужные препараты. При необходимости вирусолог модифицирует вирус через консоль эволюции.
4. **Эскалация / Карантин:** если эпидемия не сдерживается — больше заражённых, симптомы тяжелее. Безопасность организует карантин. Медики раздают аптечки.
5. **Разрешение:** эпидемия сдержана (все вылечены, иммунитет выработан), либо ситуация выходит из-под контроля.

### Разумный вирус — отдельная игровая петля

1. Спавнится как ghost-роль (или станционное событие после 45 мин).
2. Выбирает первичного пациента — заражение начинается.
3. Накапливает очки мутации, покупает симптомы.
4. Решает — тихое распространение или быстрая эскалация.
5. Медики реагируют, начинается "гонка вооружений" между вирусом и лечением.

### Взаимодействие с другими департаментами

- **Медицинский:** вирусолог — часть медблока, координирует с остальными врачами.
- **Наука:** исследует технологию Вирусологии (Tier 2, 10k очков). Сервер данных приносит пассивный доход научных очков.
- **Снабжение (Cargo):** может заказать аптечки с антибиотиками.
- **Инженерный:** может помочь с карантинными мерами (шлюзы, атмосферика).
- **Служба безопасности:** обеспечивает карантин, предотвращает намеренное распространение.

## Administrative & Server Rule Impact

### Нагрузка на админов

- Разумный вирус — антагонист с потенциалом к массовому вайпу (зомбификация).
- Намеренное распространение болезни игроком-вирусологом (грифинг) — требует тех же правил, что и намеренное причинение вреда в медблоке.

### Механические ограничения

- Зомбификация стоит 3000 очков — больше стартового запаса разумного вируса. Нужно время на накопление.
- Конкуренция штаммов позволяет медикам создать "вакцинный" вирус, дающий механическую защиту всей станции.
- Антибиотики токсичны — самолечение рискованно, что ограничивает бесконтрольное применение.
- Иммунитет после излечения предотвращает бесконечный цикл перезаражения.
- Вирус умирает в мёртвом теле (урон 5/тик), что предотвращает бесконечное существование штамма на трупах.

## Technical Considerations

### Архитектура

Система строится на стандартных паттернах ECS движка RobustToolbox:

- **Компоненты:** `DiseaseComponent`, `SentientDiseaseComponent`, `PrimaryPatientComponent`, `DiseaseContaminationComponent`, `DiseaseInfectionCloudComponent`, `DiseaseResistanceComponent`, `DiseaseImmunComponent`, `DiseaseHealingBedComponent` и др.
- **Системы:** `DiseaseSystem` (серверная, центральная), `SharedDiseaseSystem` (shared-логика: расчёт цен, проверки), `DiseaseInfectionCloudSystem`, `DiseaseContaminationSystem`, `PrimaryPatientSystem`, `DiseaseAnalyzerSystem`, `DiseaseDiagnoserSystem`, `DiseaseDiagnoserConsoleSystem`, `DiseaseEvolutionConsoleSystem`, `DiseaseSolutionAnalyzerSystem`, `DiseaseHealingBedSystem`, `DiseaseDataCollectorSystem`, `DiseaseResistanceSystem`.
- **Прототипы:** `DiseaseSymptomPrototype` — для определения симптомов в YAML. Регистрация фабрик — автоматическая через рефлексию при инициализации.
- **DiseaseData как ReagentData** — ключевое решение: вирус "живёт" внутри химической инфраструктуры, что позволяет шприцам, колбам и чашам Петри нативно работать с заболеваниями.

### Паттерн симптомов

Каждый симптом — отдельный класс с атрибутом `[DiseaseSymptom("ProtoId")]`, наследующий `DiseaseSymptomBase`. Фабричный реестр `DiseaseSymptomFactoryRegistry` создаёт экземпляры по ProtoId. Это позволяет добавлять новые симптомы без модификации существующего кода — достаточно:
1. Создать класс, наследующий `DiseaseSymptomBase`.
2. Пометить атрибутом `[DiseaseSymptom("MySymptomId")]`.
3. Добавить прототип в YAML (`diseaseSymptom`).

### Сетевое взаимодействие

- `DiseaseComponent` и `SentientDiseaseComponent` — `NetworkedComponent`, реплицируются на клиент.
- Облака инфекции синхронизируются через `ComponentGetState` (передаётся только цвет).
- Контаминация синхронизируется аналогично (цвет + уровень загрязнения).
- UI: BUI-окна для анализатора, консоли диагноста и консоли эволюции.

### Производительность

- `DiseaseSystem.Update` итерирует по всем `DiseaseComponent` каждый кадр, но фактическое обновление происходит по таймеру (1 секунда).
- `DiseaseInfectionCloudSystem` использует `EdgeSpreader` из стандартной инфраструктуры SS14 — проверенный механизм.
- Симптомы с тяжёлыми эффектами (InfectAround, EntityLookup) вызываются с интервалами 15-120 секунд.
- При большом количестве заражённых (эпидемия) возможна нагрузка от `InfectAround` (EntityLookup по MobStateComponent в радиусе). Масштаб ограничен количеством мобов на карте.
- `GetQuantityInfected` перебирает все `DiseaseComponent` — O(n), но вызывается редко (обновление UI разумного вируса).

### Зависимости

- Модуль `_Modifications/TimeWindow` — утилита для таймеров (TimedWindow, TimedWindowSystem).
- Стандартные системы SS14: Chemistry, Inventory, DoAfter, DeviceLinking, Buckle, Damage, Zombie, Popup, Chat, Interaction, Physics, Forensics, Power, Spreader.
- Текстуры: `_Modifications/Disease/` — спрайты оборудования, облаков, антаг-иконок.
- Аудио: `Audio/Effects/Diseases/`, `Audio/_Modifications/Disease/` — звуки симптомов.
- Карты: `Maps/_Modifications/Disease/` — тестовые карты или фрагменты вирусологической лаборатории.

### UI

Три BUI-интерфейса:
- **DiseaseAnalyzerWindow** — результат сканирования пациента анализатором.
- **DiseaseDiagnoserConsoleWindow** — управление лабораторией (список штаммов на сервере, кнопки действий).
- **DiseaseEvolutionConsoleWindow** — покупка/удаление симптомов и организмов.

### Текущий статус и нерешённые вопросы

- Пресет раунда `SentientDisease` и правило `SentientDiseaseRule` **закомментированы** — разумный вирус доступен только как событие (с высоким порогом: 35 игроков, 45 минут).
